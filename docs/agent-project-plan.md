# 法律小助手 Agent 项目结构与实施计划

## 1. 目标

本计划用于把通用 AI Agent 项目结构，重新设计为适合“法律小助手”的工程结构和实施路线。

当前项目定位是：面向微小企业、个体经营者、自由职业者、初创公司和普通个人，提供合同审查、合同问答、创业法律决策问答等法律风险辅助能力。

本项目不替代律师，不出具正式法律意见书。所有 AI 输出应作为初步风险提示和决策参考，并在高风险事项中明确提示咨询专业律师。

## 2. 当前仓库状态

当前仓库处于 MVP 规划阶段，已有内容包括：

- `docs/mvp-spec.md`：MVP 规格说明。
- `tasks.md`：当前任务清单。
- `legal_corpus/`：按业务方向整理的法律知识库原始资料。

当前尚未初始化前端、后端、数据库、队列、对象存储、向量库和 Agent 代码模块。

## 3. 设计原则

- 不照搬通用 Python Agent 目录，而是采用更适合当前 MVP 的 TypeScript 工程结构。
- 保留 Agent 的核心分层思想：规划、路由、技能、工具、模型、记忆、状态、日志、测试。
- 法律业务能力与通用 Agent 执行框架分离，避免后续业务扩展时互相污染。
- 法律知识库、Prompt、结构化输出 Schema 必须可追踪、可测试、可迭代。
- 第一版优先验证核心闭环，不做律师入驻、正式法律意见书、全量法规库爬取、复杂 GraphRAG、自动红线批注等能力。

## 4. 推荐技术方向

第一版沿用 `docs/mvp-spec.md` 中推荐的技术栈：

```text
前端：Taro + React + TypeScript
后端：NestJS + TypeScript
数据库：PostgreSQL
ORM：Prisma
对象存储：MinIO
缓存与任务队列：Redis + BullMQ
向量数据库：Qdrant
AI 能力：大模型 API + RAG
部署：Docker Compose
```

第一版建议先做 Web 演示端验证核心流程。等合同审查、合同问答、创业法律问答跑通后，再决定是否迁移或扩展到微信小程序。

│  ├─ api/                         # NestJS 后端服务

## 6. Agent 运行流程

```text
用户输入
→ 会话上下文加载
→ 意图识别：合同审查 / 合同问答 / 创业法律问答
→ Agent Planner 生成执行计划
→ Skill Router 选择法律技能
→ RAG 检索相关法律资料、审查规则、合同片段
→ LLM 生成结构化结果
→ Schema 校验与风险等级规范化
→ 添加免责声明和律师咨询提示
→ 返回结果并保存对话 / 审查记录
```

## 7. 核心模块边界

| 模块 | 职责 |
|---|---|
| `agent-core` | 不绑定具体法律业务，只负责任务规划、步骤执行、状态管理、错误恢复和执行日志。 |
| `legal-skills` | 承载具体法律业务技能，包括合同审查、合同问答、创业法律风险问答。 |
| `legal-rag` | 管理 `legal_corpus/`，完成文档解析、切片、向量化、召回和引用依据整理。 |
| `legal-prompts` | 集中管理法律回答原则、合同审查 Prompt、合同问答 Prompt、创业问答 Prompt。 |
| `legal-schemas` | 统一风险点、审查报告、聊天消息、知识库元数据等结构化格式。 |
| `model-client` | 统一封装大模型调用，隔离模型供应商差异，便于后续替换。 |
| `shared` | 放置跨模块共享的类型、错误码、常量、工具函数。 |
| `apps/api` | 提供上传、解析、审查、问答、报告、知识库管理等接口。 |
| `apps/web` | 提供第一版演示界面，用于验证合同审查和问答核心流程。 |

## 8. 分阶段实施计划

### 阶段 1：架构与文档

目标：先把工程边界、Agent 流程、验收标准写清楚，再进入脚手架和代码实现。

产出：

- 新增 `docs/architecture.md`。
- 新增 `docs/agent-flow.md`。
- 新增 `docs/acceptance.md`。
- 更新 `tasks.md`，让任务清单与新架构保持一致。
- 明确第一版先做 Web 演示端还是微信小程序端。

不做：

- 不初始化前后端代码。
- 不接入数据库、队列、对象存储和向量库。
- 不开始合同审查、RAG、问答等业务实现。

### 阶段 2：工程脚手架

目标：建立可运行、可测试、可扩展的基础工程。

产出：

- 初始化 monorepo。
- 初始化 `apps/api`。
- 初始化 `apps/web`。
- 配置 TypeScript、代码格式化、基础测试框架。
- 新增 `.env.example`。
- 建立基础目录和模块占位。

### 阶段 3：基础设施

目标：让后续业务模块有稳定的数据、存储、队列和向量检索基础。

产出：

- 配置 Docker Compose。
- 接入 PostgreSQL。
- 接入 Redis + BullMQ。
- 接入 MinIO。
- 接入 Qdrant。
- 建立 Prisma 数据模型草案。
- 实现健康检查接口。

### 阶段 4：合同上传与解析

目标：完成合同文件从上传到正文解析保存的闭环。

产出：

- 合同上传接口。
- 原文件保存。
- 合同记录入库。
- PDF / DOCX / TXT 文本解析。
- 合同正文保存。

第一版暂不做 OCR，只支持可复制文本 PDF、DOCX、TXT。

### 阶段 5：Agent 核心

目标：建立通用 Agent 执行机制，为合同审查、问答和创业法律问答提供统一底座。

产出：

- Agent 执行上下文。
- 任务规划器。
- Skill Router。
- 执行步骤记录。
- 错误处理和失败重试策略。
- 结构化输出校验入口。

### 阶段 6：合同审查

目标：让用户上传合同后获得结构化风险清单。

产出：

- 合同审查 Skill。
- 合同类型识别。
- 审查规则 Prompt。
- 风险等级规范化。
- 合同原文片段引用。
- 审查结果保存。

### 阶段 7：合同问答

目标：支持用户围绕已上传合同继续追问。

产出：

- 合同问答会话。
- 对话消息保存。
- 基于合同正文构建上下文。
- 回答引用合同条款片段。
- 对不确定问题提示补充信息或咨询律师。

### 阶段 8：法律知识库与 RAG

目标：把现有 `legal_corpus/` 转为可检索、可引用的知识库。

产出：

- 法律文档解析。
- 文档元数据整理。
- 文档切片。
- 向量生成。
- 写入 Qdrant。
- 相似内容检索。
- 在合同审查和问答中接入 RAG。

### 阶段 9：创业法律决策问答

目标：支持用户不上传合同，直接询问创业和经营法律风险。

产出：

- 创业法律问答 Skill。
- 合伙创业、劳动用工、客户欠款、知识产权、公司基础事务等场景识别。
- 输出风险清单、建议动作、需准备材料、律师咨询提示。

### 阶段 10：报告与验收

目标：形成可演示、可验收的 MVP。

产出：

- 审查结果详情页。
- 简版审查报告。
- 免责声明。
- 测试合同样例。
- MVP 验收清单对照。

## 9. 第一阶段的第一步应该做什么

第一步建议先做：新增 `docs/architecture.md`，把整体技术架构和模块关系定下来。

原因：

- 当前项目已有 MVP 规格，但缺少工程架构说明。
- 后续脚手架、目录、数据库、Agent 模块都依赖架构边界。
- 先写架构文档，可以避免一开始就把前端、后端、Agent、RAG 混在一起。

`docs/architecture.md` 建议包含：

- 项目目标与技术栈。
- 前端、后端、Agent、RAG、基础设施之间的关系。
- 核心服务模块划分。
- 数据流：合同上传、合同审查、合同问答、创业法律问答。
- 第一版明确不做的内容。
- 后续代码目录如何对应架构模块。

完成标准：

- 能指导后续创建 `apps/`、`packages/`、`infra/` 等目录。
- 能解释每个模块为什么存在、负责什么、不负责什么。
- 与 `docs/mvp-spec.md` 和 `tasks.md` 保持一致。
