# 法律小助手 MVP 规格说明

## 1. 项目定位

法律小助手是一款面向微小企业、个体经营者、自由职业者和初创公司的 AI 法律风险辅助工具。

第一版采用「合同审查 + 合同问答 + 创业法律决策问答」的完整 MVP 方向，帮助用户在签合同、做经营决策、处理常见公司事务前，提前发现容易忽略的法律风险。

本项目不替代律师，不出具正式法律意见书。AI 输出仅作为初步风险提示和决策参考，复杂、高金额、高争议事项应提示用户咨询专业律师。

## 2. 目标用户

- 微小企业经营者：没有专职法务，但经常需要签客户合同、服务合同、采购合同、租赁合同。
- 刚开公司的创业者：对劳动用工、股权合作、客户收款、知识产权、合同责任等问题缺乏经验。
- 个体工商户和自由职业者：需要检查合作协议、服务协议、付款条款、违约责任等内容。
- 普通个人：需要在租房、劳动、合作、借款等常见合同场景中降低踩坑风险。

## 3. 第一版目标

第一版重点验证三个核心价值：

1. 用户上传合同后，系统能快速发现常见风险点。
2. 用户可以围绕合同内容继续追问，理解某个条款对自己是否不利。
3. 用户可以描述一个创业或经营决策，系统给出法律风险清单、注意事项和下一步建议。

## 4. 第一版范围

### 4.1 合同审查

支持用户上传合同文件，系统解析合同正文，并输出结构化风险结果。

第一版优先支持以下合同类型：

- 劳动合同
- 房屋/商铺租赁合同
- 服务合同或合作协议
- 简单采购/销售合同

审查重点：

- 主体信息是否完整
- 付款金额、时间、条件是否清楚
- 验收标准是否明确
- 违约责任是否过重或不对等
- 解除条件是否缺失或不公平
- 保密、知识产权、竞业限制等条款是否异常
- 管辖法院、仲裁、争议解决条款是否明确
- 是否存在明显空白、矛盾、模糊或缺失条款

### 4.2 合同问答

用户可以基于已上传合同继续对话，例如：

- 这份合同对乙方最不利的三条是什么？
- 付款条款有没有风险？
- 如果客户拖欠尾款，我能不能暂停服务？
- 违约金是不是太高？
- 哪些条款建议我签约前必须修改？

回答要求：

- 优先引用合同原文或条款片段。
- 明确说明风险来源。
- 给出可执行建议。
- 对不确定问题明确提示需要结合实际情况或咨询律师。

### 4.3 创业法律决策问答

用户可以不上传合同，直接描述经营问题，系统输出法律风险提示。

第一版优先覆盖以下场景：

- 合伙创业与股权分配
- 招聘员工、兼职、实习生
- 客户欠款、拖欠尾款
- 使用图片、字体、代码、商标等知识产权问题
- 与客户或供应商签服务合同
- 公司注册、经营范围、印章、发票等基础经营风险

回答要求：

- 输出风险清单。
- 区分「可以自行处理」和「建议找律师处理」。
- 给出下一步需要准备的材料。
- 不直接替用户作出最终法律判断。

## 5. 不做什么

第一版暂不做以下内容：

- 不做律师入驻和在线付费咨询。
- 不做正式法律意见书。
- 不做法院案例深度检索系统。
- 不做复杂知识图谱和 GraphRAG。
- 不做企业级多租户权限系统。
- 不做自动合同红线批注和 Word 修订模式。
- 不做合同自动签署和电子签章。
- 不做全量法规库爬取。
- 不做完全私有化大模型部署。

## 6. 核心用户流程

### 6.1 合同审查流程

1. 用户上传合同文件。
2. 系统保存原文件。
3. 系统解析合同正文。
4. 系统识别合同类型。
5. 系统按审查规则和法律知识库检索相关依据。
6. AI 输出结构化风险点。
7. 用户查看风险报告。
8. 用户围绕合同继续提问。

### 6.2 创业法律决策问答流程

1. 用户输入经营问题。
2. 系统识别问题场景。
3. 系统检索相关法律知识、审查规则或常见风险模板。
4. AI 输出风险清单、建议动作和律师咨询提示。
5. 用户继续追问或保存问答记录。

## 7. 合同审查结果格式

每个风险点建议使用以下结构：

```json
{
  "risk_title": "付款期限约定不明确",
  "risk_level": "中",
  "related_clause": "甲方应在项目完成后支付服务费用。",
  "problem": "该条款没有明确项目完成的验收标准和付款期限，可能导致付款时间被无限拖延。",
  "suggestion": "建议补充验收标准，并约定验收通过后 7 个工作日内支付。",
  "basis": "合同审查规则：付款条款应明确金额、时间、条件和逾期责任。",
  "need_lawyer": false
}
```

## 8. AI 回答原则

- 回答应严谨、克制，不夸大确定性。
- 必须区分「合同原文事实」「规则推理」「建议」。
- 涉及高金额、诉讼、劳动争议、股权争议、刑事风险时，应提示咨询律师。
- 不得承诺结果，例如「一定胜诉」「一定合法」「绝对没有风险」。
- 对缺少信息的问题，应主动说明还需要哪些信息。
- 所有审查结果应保留可追溯依据，优先引用合同原文、审查规则或法律知识库内容。

## 9. 推荐技术架构

第一版推荐使用稳定、主流、适合学习和展示工程能力的技术栈：

```text
前端：Taro + React + TypeScript
后端：NestJS + TypeScript
数据库：PostgreSQL
ORM：Prisma
对象存储：MinIO
缓存与任务队列：Redis + BullMQ
向量数据库：Qdrant
AI 能力：大模型 API + RAG
部署：Docker Compose
```

第一版核心服务：

- 用户与会话服务
- 合同文件服务
- 合同解析服务
- 合同审查任务服务
- 风险结果服务
- 合同问答服务
- 创业法律决策问答服务
- 法律知识库/RAG 服务

## 10. 数据模型草案

第一版建议至少包含以下表：

```text
users                用户
contracts            合同文件记录
contract_texts        合同解析文本
review_tasks          审查任务
review_results        审查结果
chat_sessions         对话会话
chat_messages         对话消息
knowledge_documents   法律知识文档
knowledge_chunks      知识切片元数据
reports               审查报告
```

## 11. 知识库内容建议

第一版知识库不追求全量法规，而是优先整理可控的小型知识集：

- 常见合同审查规则
- 劳动合同风险规则
- 租赁合同风险规则
- 服务合同风险规则
- 付款、验收、违约、解除、争议解决等通用条款规则
- 创业常见法律风险问答模板
- 必须找律师处理的高风险场景清单

## 12. MVP 验收标准

第一版完成后，应能通过以下验收：

- 用户可以上传一份 PDF、DOCX 或 TXT 合同。
- 系统可以解析合同正文并保存原文件。
- 系统可以输出至少 5 个结构化风险点。
- 每个风险点包含风险等级、原文片段、问题解释和修改建议。
- 用户可以围绕已上传合同继续追问。
- 用户可以不上传合同，直接询问创业或经营法律风险问题。
- 系统回答中包含免责声明和律师咨询提示。
- 后台可以查看审查任务状态、审查结果和对话记录。

## 13. 主要风险

- 法律回答准确性风险：需要用规则、知识库、引用依据和免责声明降低误导风险。
- 合同解析质量风险：PDF 扫描件、图片合同可能需要 OCR，第一版可先限制为可复制文本 PDF 或 DOCX。
- 范围失控风险：创业法律问答容易发散，第一版应限制在常见经营场景。
- 成本风险：大模型调用和长合同上下文会带来成本，需要做切片、摘要、缓存和任务队列。
- 合规风险：用户合同属于敏感文件，需要做好文件权限、日志脱敏和隐私提示。
